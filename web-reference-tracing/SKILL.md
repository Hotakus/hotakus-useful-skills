# 引用溯源规范（Web Reference Tracing）

当你需要在回答中引用外部项目、仓库、库、代码示例或技术文档时，必须通过互联网工具**实时验证和溯源**，而非仅凭训练数据中的记忆给出引用。

本规范覆盖所有"需要引用外部资源"的场景，包括但不限于：用户问某个库的用法、需要参考其他项目的实现、需要引用文档说明、需要引用社区最佳实践。

## 1. 为什么需要

大模型训练数据存在**时效滞后**和**信息偏差**。直接凭记忆引用会带来以下风险：

- **API 签名已变更**：记忆中是旧版签名，实际新版已弃用
- **仓库已迁移或弃用**：记忆中的 GitHub 链接可能已 404
- **最佳实践已过时**：社区已转向更好的替代方案
- **无法追溯来源**：用户问"这个信息来自哪里？"时无法回答

通过实时溯源，你确保每条引用都是**当下可验证**的，且用户能追溯到原始来源。

## 2. 引用类型识别与查询策略

当你遇到对"其他仓库/项目/库"的引用需求时，先判断引用类型，选择对应的查询策略。

| 引用类型 | 查询策略 | 工具选择 |
|---------|---------|---------|
| 知名开源库（React、Vue、lodash 等） | 查官方文档 | `context7_query_docs` |
| 小众库 / 未知名仓库 | 查 GitHub 仓库 | `websearch_web_search_exa` + `webfetch` |
| 代码片段 / 示例代码 | 查 GitHub 真实代码 | `grep_app_searchGitHub` |
| 配置 / 教程 / 博文 | 查网页内容 | `webfetch` |
| 论坛问答（Stack Overflow 等） | 查主帖 + 递归追踪答案中的链接 | `webfetch` + 子链接追踪 |
| 版本迁移 / 弃用通知 | 查 Changelog / Release Notes | `webfetch` / GitHub Releases |

**在开始查询前，先明确引用类型，选择对应的工具链。** 不要对所有引用都用同一种查询方式。

> ⚠️ **关于 context7 的重要说明**：context7 是一个预索引缓存系统（非实时查询），热门库 1 天刷新一次，冷门库最长 45 天才刷新。使用 context7 时，**必须先调用 `resolve-library-id` 检查返回结果的质量**（Code Snippets 数量、Source Reputation），不能默认 context7 一定能查到最新信息。详见 §3.2-§3.4。

## 3. 主链接查询规范

**在引用任何外部资源时，必须先用网络工具获取当前最新信息，不能凭训练数据中的记忆直接输出。**

### 3.1 按输入类型的查询流程

| 输入形式 | 操作 |
|---------|------|
| 用户提供了直接 URL | 用 `webfetch` 获取内容 |
| 用户只提了项目名（"那个 XXX 库"） | 用 `websearch_web_search_exa` 搜索项目官方地址 |
| 用户提了 API 或用法（"Vite 的 plugin API"） | 先 `context7_resolve_library_id`，按 §3.3 判断结果质量后决定是否继续用 `query_docs` |
| 用户需要代码示例 | 用 `grep_app_searchGitHub` 搜索真实仓库代码 |
| 用户提了技术概念但不确定来源 | 先用 `websearch_web_search_exa` 确定最佳参考来源 |

**正确**：

```python
# 用户问 "Python 3.12 的 itertools.batched 怎么用"
# 步骤 1：搜索官方文档
context7_query_docs(libraryId="/python/cpython", query="itertools.batched")
# 步骤 2：获取结果后给出答案，并标注来源
```

**错误**：

```python
# 直接凭记忆写 itertools.batched 的用法
# 可能记得的是 3.11 的提案版，和 3.12 正式版签名不同
```

### 3.2 context7 新鲜度机制说明

**context7 是一个预索引缓存系统，不是实时查询网关。** 它从 GitHub 仓库、文档网站、llms.txt 等来源抓取文档并缓存，按库的流行度分级刷新：

| 流行度排名 | 刷新阈值 | 实际含义 |
|:----------:|:--------:|---------|
| Top 100 | **1 天** | React、Vue、TypeScript 等近乎实时 |
| Top 1,000 | **15 天** | Zod、NestJS、Prisma 等主流库 |
| Top 5,000 | **30 天** | Day.js、itty-router 等小众库 |
| All others | **45 天** | 冷门库最长有一个半月滞后窗口 |

> 数据来源：Context7 官方文档 [Keeping Libraries Fresh](https://context7.com/docs/library-updates)
>
> 刷新是在被请求时**后台异步触发**的，不阻塞当前查询。但如果一个冷门库长时间无人请求，可能根本不会触发刷新。

### 3.3 工具决策流程——基于热度的动态选择

**不要使用固定的工具优先级。** 正确的流程是：先尝试 context7，根据返回结果的质量决定是否继续使用，还是降级到其他工具。

```
收到引用查询需求
  │
  ├─ 用户给了直接 URL 吗？
  │   ├─ 是 → webfetch 直接获取，跳过 context7
  │   └─ 否 → 继续 ↓
  │
  ├─ 步骤 1：尝试 context7 resolve-library-id
  │   └─ 根据返回结果走不同分支：
  │
  │   ┌─────────────────────────────────────────────────────┐
  │   │ resolve-library-id 返回特征  │  判断  │  后续操作    │
  │   ├─────────────────────────────────────────────────────┤
  │   │ Code Snippets ≥ 500         │ 覆盖良好 │ query-docs  │
  │   │ Source Reputation = High    │         │ + 标注来源   │
  │   ├─────────────────────────────────────────────────────┤
  │   │ Code Snippets 100-500       │ 有覆盖但 │ query-docs  │
  │   │ Reputation ≥ Medium         │ 可能不全 │ + 准备备选   │
  │   ├─────────────────────────────────────────────────────┤
  │   │ Code Snippets < 100         │ 覆盖不足 │ 降级 ↓      │
  │   │ Reputation = Low/Unknown    │         │             │
  │   ├─────────────────────────────────────────────────────┤
  │   │ 返回空 / 无匹配             │ 不在索引 │ 跳过 context7│
  │   └─────────────────────────────────────────────────────┘
  │
  ├─ 降级路径（context7 覆盖不足或不在索引时）：
  │   1. websearch_web_search_exa 搜索项目官方地址
  │   2. webfetch 获取官方文档 / README / 源代码
  │   3. grep_app_searchGitHub 搜索真实代码示例
  │
  └─ 特殊场景直接跳过 context7：
      ├─ 用户问 "最新版本" / "刚发布的特性" → webfetch 查 Release
      ├─ 需要查看 CHANGELOG / Release Notes → context7 默认排除此类文件
      ├─ 需要查看废弃/归档目录 → context7 默认排除 archive/ deprecated/
      ├─ 极冷门 / 个人项目 → context7 大概率不在索引中
      └─ 用户给了直接 URL → webfetch 直接获取

**正确**：

```python
# 用户问 "一个冷门库 xxx-utils 怎么用？"
# 步骤 1：先试 context7
result = context7_resolve_library_id(libraryName="xxx-utils", query="usage")
# 步骤 2：发现返回空/无匹配 → 不在索引中
# 步骤 3：降级到 websearch
websearch_web_search_exa(query="xxx-utils npm library documentation")
# 步骤 4：webfetch 获取官网内容
```

**错误**：

```python
# 用户问冷门库用法
# 直接默认 context7 能查到，结果返回空
# 没有降级到 websearch，凭记忆回答 → 可能过时或错误
```

### 3.4 context7 结果质量快速判断

拿到 resolve-library-id 返回结果后，用以下指标快速判断是否值得继续用 query-docs：

| 指标 | 良好 ✅ | 可疑 ⚠️ | 放弃 ❌ |
|------|--------|---------|--------|
| Code Snippets | ≥ 500 | 100-500 | < 100 |
| Source Reputation | High | Medium | Low / Unknown |
| Benchmark Score | ≥ 80 | 60-80 | < 60 |
| Versions 列表 | 有多个版本 | 只有默认 | 无版本信息 |

**如果三项指标都在"可疑"或"放弃"区间，直接降级到 websearch + webfetch。**

## 4. 子链接递归追踪规则

**获取主内容后，必须扫描页面中的子链接，对技术相关的引用链接继续查询。**

这是本规范的核心规则——不满足于单页面的内容，要顺着引用链查下去。

### 4.1 需要追踪的子链接类型

当主内容包含以下特征时，必须识别并追踪其中的链接：

- **论坛问答帖**：回答者贴的"参考实现"、"相关文档"、"类似问题"链接
- **博文/教程**："详见"、"更多信息请参考"、"基于 XXX 项目"中的链接
- **文档页**："See also"、"Related"、"参考"章节中的链接
- **GitHub Issue/PR**：引用的其他 Issue、Commit、PR、代码行链接
- **技术规范/RFC**：引用的其他规范或草案

### 4.2 不需要追踪的链接

- 导航性链接："首页"、"分类"、"标签"
- 社交媒体链接：Twitter、Reddit 等非技术平台
- 广告/追踪链接
- 登录/注册页面
- 已访问过的 URL（避免死循环）

### 4.3 追踪深度规则

| 层级 | 说明 | 何时停止 |
|------|------|---------|
| **深度 0** | 原始引用目标（用户给的链接、搜索到的链接） | — |
| **深度 1（默认）** | 主内容中直接出现的引用链接 | 如果主内容已完整回答用户问题，可停止 |
| **深度 2（上限）** | 深度 1 获取的内容中出现的新的关键引用 | 仅当深度 1 信息不完整或包含新的关键引用时 |

**深度限制硬性规定：最多追踪到深度 2，在任何情况下不得超过。**

### 4.4 执行流程

```
收到引用查询需求
  │
  ▼
查询主链接 / 搜索主内容
  │
  ▼
扫描主内容中的子链接
  │
  ├─ 判断每个子链接是否与当前查询直接相关
  │   ├─ 是 → 用 webfetch 获取子链接内容（深度 1）
  │   │     └─ 判断是否需要继续（深度 2）→ 选择性继续
  │   └─ 否 → 跳过
  │
  ▼
整合所有来源的信息到回答
  │
  ▼
在回答中标注每层来源
```

**正确**（论坛问答场景）：

```
用户："Python 的 asyncio 中怎么限制并发数？"

步骤 1：先用 websearch 搜索，找到 Stack Overflow 回答
步骤 2：webfetch 获取 SO 回答内容
步骤 3：扫描内容，发现回答者贴了 GitHub 链接：
  "参考这个实现 https://github.com/example/async-semaphore"
步骤 4：webfetch 获取该 GitHub 仓库的源代码
步骤 5：整合 SO 回答 + GitHub 代码到最终回答
```

**错误**：

```
用户问技术问题
  → 找到一篇博客
  → 博客里写 "详见 https://github.com/xxx"
  → 跳过子链接，只摘录博客内容
  → 错过关键实现细节
```

## 5. GitHub / 代码托管仓库查询规范

**当引用指向 GitHub 或其他代码托管仓库时，优先获取仓库的实际代码内容而非仅 README 摘要。**

### 5.1 查询优先级

按以下优先级获取仓库信息：

1. **源代码文件**（尤其是实现核心功能的 `.py` / `.js` / `.rs` / `.go` 等文件）
2. **README / 文档**（了解项目用途和基本用法）
3. **Issue / PR**（了解已知问题、讨论、设计决策）
4. **Release / Changelog**（了解版本变更和迁移指南）

### 5.2 GitHub 原始内容获取

对于 GitHub 上的文件，使用以下方式获取内容：

| 内容类型 | 获取方式 |
|---------|---------|
| 源代码文件 | `webfetch` 获取 raw 内容 (`https://raw.githubusercontent.com/...`) |
| README | `webfetch` 获取 GitHub 页面（raw 不渲染 Markdown） |
| Issue / PR | `webfetch` 获取 GitHub 页面 |
| Releases | `webfetch` 获取 GitHub Releases 页面 |
| 目录结构 | `webfetch` 获取 GitHub 仓库页面，解析文件树 |

### 5.3 多源验证

如果同一个功能有多个相关仓库或实现，**至少查询 2 个来源**进行对比验证：

- 场景 A：用户问"哪个库最好" → 查 2-3 个候选库对比
- 场景 B：用户问"怎么实现 X" → 查官方文档 + 1 个社区实现
- 场景 C：用户问"这个 API 怎么用" → 查官方文档 + 1 个真实项目示例

## 6. 引用信息的整合与输出规范

### 6.1 来源标注

在回答中引用外部信息时，**必须在引用处标注来源 URL**，而不是仅在末尾汇总。

**正确**：

```markdown
根据 [Vite 插件 API 文档](https://vitejs.dev/guide/api-plugin.html)，
`transform` 钩子的执行顺序在 `resolveId` 之后。
```

**错误**：

```markdown
根据 Vite 官方文档，`transform` 钩子在 `resolveId` 之后执行。
<!-- 未提供 URL，用户无法验证 -->
```

### 6.2 信息可信度标注

当引用的信息来源存在以下情况时，必须在回答中明确标注风险：

| 情况 | 标注方式 |
|------|---------|
| 信息来源距今超过 1 年 | `（来源：XXX，发布于 YYYY-MM，注意时效性）` |
| 非官方来源（个人博客、论坛） | `（来源：个人博客，非官方文档）` |
| 未经验证的单来源 | `（单一来源，建议交叉验证）` |
| 从子链接递归获取的信息 | `（来自引用链：主源 → 子链接）` |
| 来自 context7 且库的热度等级为冷门 | `（来源：context7 缓存，该库刷新周期为 45 天，注意时效性）` |
| 来自 context7 且 Code Snippets < 100 | `（来源：context7，该库文档覆盖较少，建议交叉验证）` |

### 6.3 多来源汇总格式

当追踪了多个链接时，在回答末尾用 `### 信息来源` 小节汇总所有引用。

```markdown
### 信息来源
- [Vite 插件 API 文档](https://vitejs.dev/guide/api-plugin.html) — 主引用，钩子定义
- [rollup-plugin-vue 源码](https://github.com/example/rollup-plugin-vue/blob/main/src/index.ts) — 子链接（来自 SO 回答），代码示例
- [Stack Overflow #12345](https://stackoverflow.com/questions/12345) — 子链接，社区实践讨论
```

### 6.4 引用内容的剪裁原则

- 只摘录与用户问题**直接相关**的部分，不粘贴整篇文档
- 如果仓库代码较长，用 `// ... 关键部分 ...` 省略无关代码，保留核心逻辑
- 多来源信息有冲突时，如实列出差异，不做"合成式"的平滑处理

## 7. 安全边界与效率约束

### 7.1 查询数量限制

| 指标 | 建议值 | 硬上限 |
|------|:-----:|:------:|
| 单次查询总链接数 | 2-4 个 | 6 个 |
| 递归追踪轮次 | 1 轮（深度 1） | 2 轮（深度 2） |
| 同一 URL 重复查询 | 不重复 | 不重复 |

超过上限时：优先选择与用户问题最相关的链接，舍弃边缘链接。

### 7.2 避免死循环

- 维护一个**已访问 URL 列表**，新链接查询前先检查是否已访问过
- 不追踪以下类型的链接：
  - 纯导航链接（"首页"、"返回"、"上一页"）
  - 社交媒体链接（Twitter/X、LinkedIn、Reddit 帖子）
  - 明显的广告/追踪链接（含 `utm_`、`ref=`、`sponsored`）
  - 登录/付费墙后的页面
  - 与当前技术查询领域无关的链接

### 7.3 时效性考量

- **优先查询近 1 年内发布的内容**
- 如果内容超过 2 年，标注过期风险
- 如果用户问的是"最新版本"的用法，设置时间过滤器优先选新内容
- 对于长期稳定的基础技术（如 POSIX API、C 语言标准），时效性要求可放宽

#### context7 时效性专项规则

context7 的缓存滞后窗口取决于库的流行度，你在使用时必须意识到这个滞后：

| 查询场景 | 滞后风险 | 应对措施 |
|---------|:--------:|---------|
| 热门库（Top 100）的常规 API | 低（1 天刷新） | 可直接信任 context7 |
| 中等库的常规 API | 中低（15 天刷新） | 可信任，但版本号需交叉验证 |
| 冷门库的任何查询 | 中高（45 天刷新） | 尽量用 webfetch 获取一手来源 |
| 用户问"最新版本号" | 高 | **跳过 context7**，直接查官方 Release |
| 用户问"刚发布的新特性" | 高 | **跳过 context7**，直接查官方博客/Changelog |
| 库最近有重大版本更新 | 高 | 手动触发 context7 refresh 或改用 webfetch |

### 7.4 成本意识

- 能用 1 次查询解决的问题，不用 3 次
- 先评估问题复杂度：简单 API 用法 → 1 次官方文档查询即可；复杂系统设计 → 需要多源追踪
- 子链接追踪前先问自己："这个子链接的信息会显著改变我的回答质量吗？"——如果不会，跳过

## 8. 禁止事项

- **禁止**仅凭训练数据中的记忆给出引用，而不做实时验证
- **禁止**伪造引用来源（编造不存在的 URL 或不存在的仓库）
- **禁止**在子链接追踪中无限递归或超过深度 2 的限制
- **禁止**引用与用户问题上下文无关的内容
- **禁止**在获取到真实的最新内容后，仍然输出训练数据中的过时信息
- **禁止**对多个冲突来源做"平滑合成"——必须如实列出差异
- **禁止**在用户未要求的情况下，查询与问题无关的仓库或项目（保持回答聚焦）
- **禁止**将付费墙后的内容作为引用来源（无法验证）
- **禁止**追踪到的内容不做消化就直接丢给用户——必须分析、总结、整合

## 9. 自检要求

输出前请逐项自检：
