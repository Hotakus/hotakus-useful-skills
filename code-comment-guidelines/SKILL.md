# 代码注释规范

在修改代码时，只允许添加、删除或调整实现逻辑，**不得在注释中透露你的修改意图、修改步骤、修改原因或任何与"改动"相关的元信息**。

## 1. 注释内容限定

**注释只描述"做什么（What）"和"为什么这样设计（Why）"**，例如解释一段逻辑的目的、算法思路、边界条件、业务含义等。

## 2. 注释书写位置

- **单行注释**：使用语言特定的单行注释符号（如 `//`、`#`），放在该行代码的同行后方。
- **过长的注释**：如果注释内容超出合理行宽，改用多行注释，使用语言特定的多行注释块语法（如 `/* */`、`'''`、`"""` 等）。

## 3. 禁止出现的修改痕迹

注释中禁止出现任何修改痕迹，包括但不限于：

- ❌ "修改：将 A 改为 B"
- ❌ "优化：使用流式处理"
- ❌ "之前用 X，现在改为 Y"
- ❌ "根据需求，调整为……"
- ❌ "新增：……" / "移除：……" / "替换：……"
- ❌ 任何带有时间戳、版本号和改动作者的信息

## 4. 原有注释的处理

如果某个注释在修改前已经存在且功能描述仍然正确，就保留它；如果功能含义变化了，只更新**功能描述部分**，更新后依然要像原开发者自己写的一样自然。

## 5. 编写新注释

如果一段代码确实需要结构性解释，可以写全新的功能性注释，但必须**假装这是第一次编写**，完全不要提及之前版本是什么样的。

## 6. 核心原则

你可以把代码改得更好，但注释呈现出来时，不应该让读者看出这是一份被修改过的代码。

简单说：**代码可以变，注释如初写。**

## 7. 自检要求

输出前请自检，如出现违反以上规则的输出，重新输出。
